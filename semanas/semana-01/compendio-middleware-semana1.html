Conceptos de Sistemas Distribuidos

Evolución de los Sistemas Distribuidos

Definición Formal

"Un conjunto de computadores independientes, interconectados a través de una red y con capacidad de colaborar con el fin de realizar una tarea."

Académicamente: "Una red de ordenadores independientes que aparecen como un sistema coherente a sus usuarios"

Ventajas de los Sistemas Distribuidos

Compartición de Recursos

Reflejo de la arquitectura organizacional moderna, permitiendo compartir recursos a través de la red

Escalabilidad

Incrementar recursos según demanda, superando limitaciones monolíticas

Tolerancia a Fallos

Replicación de recursos para asegurar disponibilidad ante fallos

Roles del Middleware

Definición y Propósito

Middleware (Mw): Capa de software que establece conexión entre el sistema operativo y las aplicaciones, datos y usuarios.

Función Principal

Proporcionar servicios y funciones comunes como Single Sign-On (SSO) o gestión de APIs

Funciones Clave del Middleware

1. Abstracción y Conectividad

El middleware proporciona un conjunto de interfaces de programación de aplicaciones (APIs) "más funcional que el sistema operativo y los servicios de red", permitiendo transparencia de localización, independencia de servicios de red y escalabilidad

2. Integración y Heterogeneidad

El middleware es fundamental para "soportar la comunicación entre procesos a través de plataformas heterogéneas". Las herramientas de integración conectan sistemas mediante transformación, mensajería empresarial y autenticación SSO

3. Desacoplamiento

Opera como "espacio de memoria compartida" permitiendo desacoplamiento temporal y referencial

Tipos de Middleware

El middleware puede tomar diversas formas según las necesidades de comunicación y procesamiento en sistemas distribuidos

Middleware como Integrador de Capas

Middleware Transversal

El middleware no solo conecta capas, sino que proporciona servicios transversales que atraviesan toda la arquitectura

Middleware de Comunicación

  • gRPC: Entre microservicios
  • WebSockets: Tiempo real UI-Backend
  • Kafka/RabbitMQ: Mensajería asíncrona

Middleware de Seguridad

  • JWT/OAuth2: Autenticación entre capas
  • TLS: Cifrado de comunicaciones
  • API Gateways: Control de acceso

Estilos y Capas Arquitectónicas

Arquitecturas Multicapa

La arquitectura de una aplicación puede referirse a sus capas lógicas (layers), que son "capas de software que nosotros, los desarrolladores/as, creamos". El principio es dividir una aplicación en capas que colaboran mediante interfaces bien definidas

— DWES DOCS, IES Celia Viñas

El diseño más utilizado actualmente es el diseño en tres capas

Ventajas del Diseño Multicapa

  • Separación de responsabilidades
  • Mantenibilidad y escalabilidad
  • Reutilización de componentes
  • Testabilidad independiente

Puntos de Integración entre Capas

Conectividad y Comunicación

Los puntos de integración definen cómo las capas se comunican e intercambian información de manera controlada

Presentación ↔ Negocio

  • APIs REST: HTTP/JSON para operaciones CRUD
  • GraphQL: Consultas flexibles de datos
  • WebSockets: Comunicación en tiempo real
  • RPC/gRPC: Llamadas a procedimientos remotos

Negocio ↔ Datos

  • ORM/ODM: Mapeo objeto-relacional
  • SQL/NoSQL: Consultas directas a BD
  • Stored Procedures: Lógica en base de datos
  • Connection Pools: Gestión de conexiones

Capas vs Niveles

Capa (Layer)

Referencia lógica

Segmentación desde punto de vista lógico (Presentación, Lógica, Datos)

Nivel (Tier)

Referencia física

Distribución física en hardware (ej: 3 capas en 1 máquina = 3 capas, 1 nivel)

Patrón Modelo-Vista-Controlador (MVC)

Caso particular de arquitectura 3 capas con responsabilidades estandarizadas

Modelo

Lógica de negocio, acceso a datos, filtros y restricciones del sistema

Vista

Interfaz de usuario, obtiene datos del Modelo para generar la interfaz

Controlador

Gestiona eventos, accede al Modelo y delega a la Vista

Estilos Arquitectónicos Modernos

Cliente-Servidor (C-S)

Modelo popular en aplicaciones web. Servidor espera peticiones pasivamente, cliente invoca y espera respuestas

Peer-to-Peer (P2P)

Procesos con capacidades idénticas, interacciones directas. Descentralización, distribución, balance de carga

Microservicios

Separación en módulos independientes. API Gateway como punto de entrada, escalabilidad y resiliencia
